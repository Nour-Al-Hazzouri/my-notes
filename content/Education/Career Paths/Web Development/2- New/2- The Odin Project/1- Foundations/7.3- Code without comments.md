### Coding Without Comments - Summary

This article argues that code comments are widely misunderstood and frequently abused, often becoming a "crutch" for poorly written code. The main philosophy is that developers should **write code as if comments didn't exist**, forcing them to create the simplest, most self-documenting code possible.

---

### The Problem with Excessive or Redundant Comments

When comments are used incorrectly, they become excess that hinders readability, such as:

- **Bloat and Redundancy**: Writing extensive documentation (like author names, SSN, eye color, or the entire _algorithm_) when the code already shows _how_ it works.
    
- **Misunderstanding**: Developers often use comments to tell the reader _how_ the code functions when they should be using the code to tell that story and the comments to explain **why** (context or rationale).
    

---

### The Power of Self-Documenting Code

The recommended solution is to use **refactoring** to eliminate the need for explanatory comments.

|**Problematic Code (Opaque Intent)**|**Solution (Self-Documenting Code)**|
|---|---|
|**Code Block:** A mysterious loop of math functions (`r = n / 2; while ( abs( r - (n/r) ) > t ) { ... }`)|**Refactoring:** Wrap the block in a function with a descriptive name (`private double SquareRootApproximation(n)`).|
|**Result**|The code's purpose is now perfectly understandable **without adding a single comment**.|

When you find yourself wanting to add a comment to explain _what_ a block of code does, you should first try to rewrite, refactor, and rearchitect that code until it is straightforward and obvious.

### Comments as a Last Resort

Comments should only be added:

1. **As a method of last resort**, after all refactoring possibilities have been exhausted.
    
2. **To explain _why_** a certain unintuitive implementation was chosen, or to point out external context (as detailed in the previous article, _Code Tells You How, Comments Tell You Why_).
    

### Junior vs. Senior Developers

- **Junior Developers** often rely on comments to tell the story of the code.
    
- **Senior Developers** rely on the **code to tell the story**, using comments only as "narrative asides."
    

The author concludes that writing good, meaningful comments is hard and requires strong writing skills. If code feels too complex to understand without comments, the code itself is likely flawed and needs simplification.
