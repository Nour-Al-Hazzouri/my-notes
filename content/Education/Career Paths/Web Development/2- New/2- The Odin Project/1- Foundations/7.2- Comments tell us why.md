### Code Tells You How, Comments Tell You Why - Summary

This blog post by Jeff Atwood explores the proper philosophy behind writing code comments, emphasizing that the primary goal should be to make code self-explanatory.

---

### 1. Code is Written for People, Not Machines

The fundamental principle, cited from _Structure and Interpretation of Computer Programs_ and Donald Knuth's _Literate Programming_, is that:

> "Programs must be written for people to read, and only incidentally for machines to execute."

Developers should prioritize explaining their intent to human beings, first by writing code that is simple, concise, and clear, and second by using carefully chosen, meaningful variable names. When writing code, **always keep your audience (other programmers) in mind**.

### 2. The Best Comments Are the Ones You Don't Need

You should first strive to simplify your code until comments are no longer needed as a "crutch."

- **Refactoring before Commenting**: If you feel the need to write a comment explaining _how_ a section of code works, your first response should be to **refactor** the code to make it simpler and more readable.
    
- **Meaningful Names**: Replacing cryptic expressions and single-letter variables with well-named constants and functions often eliminates the need for explanatory comments.
    
    - **Example**: Replacing a complex conditional with `int oldX = deviceInfo->position.x; int directionCode = (x > oldX) ? DIRECTIONCODE_RIGHT : ...` is clearer than the original complex block _even with_ in-line comments.
        

### 3. The Role of Comments: Explaining the _Why_

Despite efforts to simplify, code alone has limits and cannot be entirely self-documenting. **Comments are essential for providing context and rationale.**

- **Code tells you _how_ the program works.**
    
- **Comments tell you _why_ it works.**
    

Comments should address the context, rationale, and history behind certain choices that are not obvious from the code itself.

- **Example of a Good Comment**: Explaining a complex design choice that goes against intuition:
    
    > `/* A binary search turned out to be slower than the Boyer-Moore algorithm for the data sets of interest, thus we have used the more complex, but faster method even though this problem does not at first seem amenable to a string search technique. */`
    
- **Simple Example**: For a complex or idiomatic language construction (like Perl's string reversal), a simple comment clarifying the action is helpful: `# Reverse the string`.
    

**Key Takeaway**: Don't shortchange your fellow developers. Ensure your code explains **how** and your comments explain **why**.
