### 10 Principles for Keeping Your Programming Code Clean - Summary

Clean code is crucial for **teamwork** and future maintenance, as different people (or your future self) will need to work with what was previously written. These principles aim to make code transparent and readable.

---

### The 10 Principles for Clean Code

#### 1. Revise Your Logic Before Coding

Before starting to code, use **flow diagrams or written pseudocode** to verify the logic behind complex functionality. Getting the logic right faster avoids messy code replacements and additions later.

#### 2. Clearly Expose the Structure of the Page

Use main containers with **representative and descriptive IDs** (e.g., `<div id="header">`, `<div id="footer">`). This makes the page structure evident and transparent for anyone adding or removing content later.

#### 3. Use the Correct Indentation

Indentation distinctly displays the opening and closing points of each element. If lines are glued to the left, the structure is lost, making it hard to find where elements close. Consistency is key to supporting a clear structure.

#### 4. Write Explanatory Comments

Comments are an effective way of **code documentation**, providing quick, straight-to-the-point information right where it is needed.

#### 5. Avoid Abusing Comments

Comments should be there for a reason and not:

- Writing explanatory notes to self (e.g., `/* Will finish this later… */`).
    
- Blaming others (e.g., `/* John coded this. Ask him. */`).
    
- Writing vague statements (e.g., `/* This is another math function. */`).
    
- Leaving large chunks of **commented-out code** after development, as this creates terrible confusion and should be deleted if unused.
    

**Good Examples of Comments:**

- Authoring specifications (e.g., `/* Coded by John, November 13th 2010 */`).
    
- Detailed statements on a method's functionality (e.g., `/* This function validates the login form with the aid of the e-mail check function */`).
    
- Quick notifications or labels stating a recent change (e.g., `/* Added e-mail validation procedure */`).
    

#### 6. Avoid Extremely Large Functions

As functionality grows, methods tend to become confusingly large (e.g., hundreds of lines). It is a better practice to **break up large functions into smaller ones**. This can also help the team reuse repeated procedures via separate functions.

#### 7. Use Naming Standards for Functions and Variables

The name of a variable or function should be **descriptive enough** to give a general idea of its purpose. Whether following company-established standards (e.g., prefixing numeric variables with `int_`) or setting your own, respecting these standards saves time during future redesigns.

#### 8. Treat Changes with Caution

Adding, removing, or changing code requires awareness of not disrupting previous efforts to keep the code clean and ordered. This involves:

- Maintaining the **correct indentations** (e.g., augmenting content inside an `IF` clause).
    
- **Commenting on the modification** made or broadening existing comments.
    
- Respecting established standards.
    

#### 9. Avoid Indiscriminate Mixing of Coding Languages

Mixing languages, such as using in-line CSS styling or scattered JavaScript tags throughout the HTML structure, is discouraged. This results in huge element tags with embedded styles and interruptions to the code's flow, leading to confusion. Having appropriate divisions between languages brings order to the applied logic.

#### 10. Summarize Your Imports

While importing additional coding languages from different files is better than mixing, avoid abusing this. If there are too many style sheets, summarize them into one or two. This not only saves space and tidiness but also **saves loading time**, as each imported file is an HTTP request that affects application performance (and avoids issues like Internet Explorer's stylesheet limit).

---

### Conclusion

Convenience often leads to disregarding coding efficiency. It is highly necessary to keep up with standards to maintain code that is readable in the future. Code should be sufficiently open and understandable to truly support teamwork.
